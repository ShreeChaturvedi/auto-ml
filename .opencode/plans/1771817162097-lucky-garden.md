# Plan: Data Upload Tab Redesign — LLM-Driven Onboarding Flow

## Overview

Transform the upload tab from a static file-drop zone into a multi-stage, LLM-guided onboarding experience. The LLM becomes involved from the very first phase — understanding uploaded data, asking clarifying questions via structured UI (the `ask_user` tool), and generating a project plan that becomes the north star for all subsequent phases.

## Architecture: Stage Machine

The upload phase becomes a **state machine** with 5 stages, tracked in `project.metadata.uploadStage`:

```
UPLOAD  →  PROCESSING  →  INTENT  →  PLANNING  →  REVIEW  →  (auto-advance to data-viewer)
```

- **UPLOAD**: Clean file upload only (no CustomInstructions). Inline-editable title/description in header.
- **PROCESSING**: Compute animation SVG plays (min 3s). Backend profiles data in parallel. Results appear on right side of animation.
- **INTENT**: User describes their goal (smart textarea + suggestion chips). Submits to LLM.
- **PLANNING**: LLM streams — uses data tools, calls `ask_user` for structured questions (up to 5 rounds), then generates markdown plan.
- **REVIEW**: Rendered markdown plan. User can approve, directly edit, or chat to request changes.

User can navigate back to UPLOAD stage to add/remove files (which resets to PROCESSING on next "Next" click).

```ts
const onboardingSchema = z.object({
  projectId: z.string().min(1),
  userIntent: z.string().optional(),         // User's goal description

  toolResults: z.array(toolResultSchema).optional(),  // Prior tool results

```

**Handler logic**:
- Load all datasets and documents for the project (metadata only — use `list_project_files` equivalent)
- Load RAG snippets if documents exist
- Build the onboarding prompt (see 1C)
- Stream via `streamLlmResponse` with `kind: 'onboarding'`
- Handle `ask_user` tool calls specially in the envelope: wrap as `{ type: 'envelope', envelope: { version: '1', kind: 'onboarding', ask_user: <questions> } }`

**Critical**: The `ask_user` tool call should NOT go through the normal tool_calls envelope path. It needs its own envelope field so the frontend can distinguish it from auto-executable tools:

In the `onToolCall` handler within `streamLlmResponse`, add a case:
```ts
if (call.name === 'ask_user') {
  // Store as askUserEnvelope instead of adding to toolCalls
  askUserPayload = call.args;
  return;
}
```

Then in the post-stream envelope emission:
```ts
if (askUserPayload) {
  writeEvent({
    type: 'envelope',
    envelope: {
      version: '1',
      kind: 'onboarding',
      ask_user: askUserPayload,   // New field
      tool_calls: toolCalls.length > 0 ? toolCalls : undefined  // May also have data tool calls
    }
  });
}
```

Orchestrates:

2. Renders the `ComputeAnimation` SVG component
3. Collects processing results and feeds them to the animation's right side
4. Ensures minimum 3-second animation duration via `Promise.all([dataPromise, minDelayPromise])`
5. On complete, calls `onComplete(results)` to advance to intent stage

```tsx
export function ProcessingStage({ projectId, onComplete, onBack }: ProcessingStageProps) {
  const [results, setResults] = useState<ProcessingResult[]>([]);
  const [isComplete, setIsComplete] = useState(false);

  useEffect(() => {
    const process = async () => {
      const minDelay = new Promise(r => setTimeout(r, 3000));
      const dataPromise = gatherProcessingResults(projectId); // Collects file metadata from store
      const [data] = await Promise.all([dataPromise, minDelay]);
      setResults(data);
      setIsComplete(true);
      // Brief pause for final result card to animate in
      await new Promise(r => setTimeout(r, 800));
      onComplete(data);
    };
    process();
  }, [projectId]);

  return (
    <div className="flex h-full items-center justify-center">
      <ComputeAnimation files={projectFiles} results={results} isComplete={isComplete} />
    </div>
  );
}
```

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| LLM doesn't call `ask_user` tool reliably | Strong prompt engineering + fallback: if LLM skips questions, go straight to plan generation |
| LLM generates malformed `ask_user` args | Validate with zod on backend before sending to frontend; show error gracefully |
| Animation feels janky on low-end devices | Use CSS animations (GPU-accelerated) over JS; provide reduced-motion fallback |
| Plan markdown is inconsistent | Enforce structure in prompt; parse sections on frontend for validation |
| Stage persistence issues | Default to 'upload' if stored stage is invalid; clear stage on file changes |
| question round loop doesn't terminate | Hard cap at 5 rounds enforced on both frontend and backend |
| Large plans exceed metadata size limits | Set generous limit (50KB) for projectPlan field |
